IP: Summative Assignment: Fault Injection - Lecture fi

Introduction

This is an exercise in understanding compile-time and run-time errors and their causes
  • Software fault injection is a technique for analysing the effectiveness of testing
  • Can also be used for assessing fault tolerant systems
  • Mutation testing is a kind of fault injection where changes are made to source code
  • We are going to use mutation injection and see if the changes are identified by
    • Compiler
    • Run-time system
    • Test results
We will test two hypotheses:
  • The Java type system helps to identify programming errors at compile-time instead of run-time
  • It is difficult for compilers to identify correctly what progamming errors have been made and where

Mutations to consider

Use a random number generator to identify which file/line/character/variable is affected.
  • Remove a line
  • Change a character
  • Replace variable

Code to work on

The code is a student solution to a programming exercise with five classes
  • Track representing music tracks
  • Artist representing music artists (bands and soloists)
  • Album contains a list of tracks
  • Library contains a whole music colllection
  • Main includes a main method which allows details to be loaded from a file
Check that you can compile the files and call the main method from the class Main.

Remove a line

The first task is to remove a line at random from one of the files
  • Select a class at random (using java.util.Random)
  • Select a line at random from that file
  • Remove that line from the file (by hand, using computer-generated class and line)
Then we need to see if any problem is identified, and if so, what and where. We will use the same process for all three tasks
  • Recompile all the classes
  • If there is a compilation error then record
    • the error message
    • the line it refers to
    • your opinion as to whether the error message has correctly identified the problem (just a yes/no)
    • how close the error message was in getting the line on which the problem occurred (a number representing how many lines out the message was)
  • If there is no compilation error then execute the main method
  • If there is a run-time error record then record details of the message and line number as before
  • If there is no error message then see if the main method executes as before
  • Record whether, in your opinion, the test has correctly identified any problem with the code
Repeat the process 10 times and collate the results.

Change a character

The second task is to change a character at random in one of the files
  • Select a class at random (using java.util.Random)
  • Select a line at random from that file
  • Select a character at random from that file
  • Change (by hand) that character to a random character in the range
    [a-z][A-Z][0-9][,.;:(){}=]
  • Hint: use charAt from the String class
  • Record any errors as before
Repeat the process 10 times and collate the results.

Change a variable

The third task is to change a variable name at random in one of the files.
Before you start label all of the variables with a number so you can select them at random. Include
  • Instance variables (i.e. fields)
  • Local variables
  • Formal parameters
  • Variables introduced in for-loops
You also need to label every occurrence of a variable being used including
  • In an assignment (on either side)
  • In an method call
  • In an actual paramater
  • In the specification of a collection for a for-each loop
Now you can carry out the fault injection as follows:
  • Select a class at random (using java.util.Random)
  • Select a variable occurrence at random from that file (i.e. from the second list)
  • Replace (by hand) that occurrence with another variable chosed at random from that class (from the first list)
  • Record any errors as before
Repeat the process 10 times and collate the results.
